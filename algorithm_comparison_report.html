Executive Summary

The biggest distinction is independence. QF treats each donor as an independent signal. COCM tries to measure whether donors are connected through shared project affinities and only rewards cross-cluster signal. In this repo's UI, the cluster signal is usually the donation matrix itself, so projects double as clusters.

Harsh=True is the default for current COCM. It changes COCM from "soften close-cluster contributions toward square-root treatment" into "remove close-cluster contribution from that cluster direction." In code, it replaces K = k * sqrt(c) + (1 - k) * c with K = (1 - k) * c.

Fancy=True is not a current UI toggle. In this repo it is a historical name for the non-harsh/legacy COCM scaling formula: K = k * sqrt(c) + (1 - k) * c. The closest current setting is harsh=False, while Legacy COCM is documented as the old fancy=True replica.

Question Short answer
What is QF? Quadratic funding over individual donors: many independent small donors can beat one large donor.
What is COCM? Cluster-aware matching: it rewards support that crosses distinct donor/project affinity clusters and discounts coordinated or tightly connected support.
What is COCM og? A current COCM variant using a broad boolean "connected to this cluster" indicator from a weighted friendship matrix.
What is Legacy COCM? A replica of the older/original COCM implementation: pct-friends style indicators, fancy/non-harsh scaling, legacy alignment, and no harsh mode.
Which algorithms does harsh affect? COCM, COCM og, COCM pct_friends, plus the COCM leg of half-and-half and pctCOCM. It does not affect QF, pairwise, donation_profile_clustermatch, or Legacy COCM.

Common Pipeline

Most algorithms return a raw matching signal per project. get_qf_matching() then reserves any configured minimum matching floor, normalizes the residual algorithmic signal, caps, and scales that signal into matching dollars.

raw_signal_by_project -> reserve min_matching_floor * project_count from matching_pool -> normalize raw signal over the residual pool -> add min_matching_floor to each project -> apply matching cap with iterative redistribution -> matching_percent = capped_share * 100 -> matching_amount = capped_share * matching_pool

Raw signal is not direct donations

Functions such as standard_qf() and COCM() return the match-only signal. The comment in fundingutils.py says direct donations must be added separately if total project funding is needed.

The cap is shared by all algorithms

check_matching_cap() clips projects above the cap and redistributes excess proportionally to projects below the cap until no project is over cap. The cap is applied after any minimum floor has been added.

Minimum matching floor

min_matching_floor is a per-project dollar lower bound. If the requested floor does not fit in the pool, the effective floor is scaled down evenly. If the floor is above the per-project cap, the cap wins and total distributed matching can be below the full pool.

Cluster input in the UI

The CSV calculator passes cluster_df=donation_matrix. That means a donor's project donation profile is also treated as the donor's cluster membership profile.

What Harsh=True Means

In current COCM, harsh only changes the K matrix, which is the per-donor, per-cluster scaled contribution used for a target project.

harsh=False

K(i,g) = k(i,g) * sqrt(c_i) + (1 - k(i,g)) * c_i

A donor close to cluster g is moved toward square-root treatment. A donor far from cluster g stays closer to linear contribution. This is the gentler, legacy/fancy-style behavior.

harsh=True

K(i,g) = (1 - k(i,g)) * c_i

A donor fully close to cluster g contributes zero in that cluster direction. A donor unrelated to g contributes the full linear amount. Partial closeness creates a proportional discount.

Case k(i,g) harsh=False harsh=True Interpretation
Donor is directly in the cluster 1 sqrt(c_i) 0 Harsh mode removes same-cluster signal instead of softening it.
Donor is partly close to the cluster Between 0 and 1 Blend between c_i and sqrt(c_i) Proportional discount of c_i Harsh mode treats proximity as a penalty.
Donor is unrelated to the cluster 0 c_i c_i Both modes preserve far-cluster signal.

Real-world meaning: Harsh=True asks for independence more aggressively. It says that support from the same or nearby cluster should not create matching signal in that cluster direction. Projects backed by tight, highly overlapping communities tend to lose relative share; projects backed by donors whose profiles bridge distinct communities tend to gain relative share.

Because final results are normalized to the matching pool, harsh mode usually changes relative allocation, not the total amount distributed.

What Fancy=True Means

Fancy is a historical COCM setting, not a live parameter in the current UI. The repo still refers to it in comments and in legacy_COCM(), whose docstring says it is an exact replica of legacy COCM with fancy=True.

QF vs COCM

Technical Difference

QF operates per project and per donor. It sums square roots of contributions and subtracts direct donations, which creates extra matching when many donors give to the same project.

COCM operates per target project, per donor, and per cluster. It builds a donor-cluster proximity matrix k_indicators, scales target-project contributions into K, projects those through normalized cluster memberships into P_prime, symmetrizes with sqrt(P_prime(g,h) * P_prime(h,g)), and sums only off-diagonal cluster pairs.

Real-World Difference

QF asks: "How many people supported this project, and how strongly?" It works best when donor choices are independent.

COCM asks: "Do these supporters represent independent communities, or are they one coordinated cluster?" It is meant to reduce the advantage of collusive groups, tightly coordinated fans, or many wallets that all share the same support pattern.

Dimension QF COCM
Unit of independence Individual donor Donor's position across clusters/project affinities
Main risk it addresses Underfunding public goods with broad small-donor support Coordinated or correlated donor groups gaming QF assumptions
What gets boosted More donors to the same project Cross-cluster, reciprocal support patterns
What gets discounted Nothing about donor similarity Same-cluster or close-cluster support, especially with harsh=True
Potential false negative Can over-reward coordinated wallets Can under-reward legitimate niche communities whose supporters naturally overlap

Pre-processing settings that change every algorithm's input

  • min_donation_threshold_usd removes small donations before the matrix is built.
  • min_passport_score and min_model_score use OR logic in the CSV calculator when both are present: a donation is kept if it passes either threshold.
  • Self-votes are dropped when recipient/payout addresses are available.
  • The CSV calculator aggregates duplicate donor/project rows with sum for all algorithms except Legacy COCM, which uses mean.
  • matching_pool_usd does not affect ranking before caps; it scales final dollars after shares are computed.

Important Edge Cases and Caveats

  • Legacy COCM ignores the Harsh toggle. The CSV UI enables the toggle for Legacy COCM, but get_qf_matching() routes legacy to legacy_COCM() before any harsh-aware branch.
  • Legacy COCM can differ from current pct_friends even with harsh=False. The algorithmic core is similar, but alignment, duplicate aggregation, and zero-column handling can change outputs.
  • All COCM methods drop diagonal cluster-pair signal. Same-cluster signal is not counted; only off-diagonal reciprocal cluster-pair signal becomes raw match.
  • Single-project and all-project donor patterns can be weak under COCM. The legacy app explicitly warns that COCM can ignore or reduce these patterns as a by-product of the cross-cluster logic.
  • Raw COCM signal is not final dollars. The checked-in legacy trace for real_sampled.csv has total raw COCM signal 126.16179064013517, then normalizes, caps at 20%, and scales to a $50,000 matching pool.
  • Runtime comparison was not regenerated here. The local bare python3 did not have pandas installed, so this report relies on source inspection and the checked-in trace rather than newly computed sample tables.

Frequently Asked Questions

1. Why do single-project donors often receive little or no matching signal under COCM?

COCM only rewards cross-cluster support. A donor who only supports one project contributes only to that project’s own cluster direction. When the diagonal of the P matrix is dropped, that same-cluster signal is removed. Only donations that bridge different clusters generate matching.

2. What is the main difference between standard QF and COCM?

QF treats every donor as fully independent and applies quadratic scaling per project. COCM builds a model of donor–cluster proximity (using co-donation patterns) and discounts contributions that stay inside the same cluster, rewarding only the cross-cluster portion of support.

3. Does turning on harsh=True change the total size of the matching pool?

No. Harsh mode only changes the relative shares among projects. After normalization the total is always 100% of the pool (subject to the cap). It simply makes the penalty for same-cluster support stricter.

4. Why does the Legacy COCM algorithm ignore the harsh toggle in the UI?

get_qf_matching() routes “Legacy COCM” to the separate legacy_COCM() function, which hard-codes the original non-harsh (fancy) scaling formula and never inspects the harsh parameter.

5. How does the CSV calculator decide what the clusters are?

It passes the donation matrix itself as cluster_df. Each donor’s pattern of projects they supported becomes their cluster membership vector. Projects therefore serve as both the funding targets and the affinity clusters.

6. What does the raw signal returned by standard_qf() or COCM() represent?

It is the matching-only portion before normalization, capping, and scaling to the actual dollar pool. You must add the original direct donations if you want a project’s total funding figure.

7. Why is the diagonal of the P matrix always removed in COCM calculations?

The diagonal represents same-cluster (self-to-self) signal. COCM’s design goal is to subsidize mutual support between different clusters. Same-cluster contributions are deliberately excluded.

8. How does the matching cap work when a project exceeds the percentage limit?

check_matching_cap() iteratively clips the over-cap project to the limit and proportionally redistributes the excess percentage to all projects still below the cap, repeating until no project exceeds the cap. The cap runs after min_matching_floor is added, so a low cap can reduce a project's final match below the requested floor.

8a. How does min_matching_floor affect pool accounting?

The allocator reserves project_count * min_matching_floor first, runs the selected algorithm on the remaining pool, then adds the floor to each project before applying caps. If the requested floor is too large for the pool, the effective floor is scaled to matching_pool / project_count and the residual algorithmic pool becomes zero.

9. What are the three calcstyle options for COCM and how do they differ?

markov uses a Markov-style random-walk proximity (PJP) to compute soft cluster membership. og uses a boolean “reachable through any friend” indicator. pct_friends uses the fraction of a donor’s friends that belong to each cluster (maxed with direct membership).

10. When would you choose pctCOCM over half-and-half?

pctCOCM lets you dial any mixture (0 = pure QF, 1 = pure COCM). half-and-half is just the special case of 50/50. Use pctCOCM when you want a precise blend other than exactly 50%.

11. What does donation_profile_clustermatch do that standard QF does not?

It first groups donors who have identical binary support patterns into the same cluster, then runs quadratic funding over those clusters instead of individual donors. Many identical small donors are treated as one voice.

12. How is the pairwise coefficient k_i,j computed?

k_i,j = M / (M + dot(sqrt(profile_i), sqrt(profile_j))) with M = 0.01. Donors with very similar overall donation profiles receive a small coefficient; dissimilar profiles receive a coefficient close to 1.

13. What is TQF in Home.py and how does it relate to the current algorithms?

TQF applies an arbitrary per-donor scale_factor (token weighting) to each row of the donation matrix and then falls back to standard QF. It is a legacy scaling path, not a distinct matching formula.

14. Can any COCM variant ever distribute more total matching dollars than standard QF?

No. All algorithms ultimately normalize their raw signal to sum to 1.0 before scaling to the matching pool. Differences are only in which projects receive larger or smaller shares.

15. Why can Legacy COCM give different numbers from current COCM pct_friends even with harsh=False?

Legacy uses older alignment logic (does not drop zero-sum projects/clusters), different aggregation of duplicate donors, and fill_diagonal instead of trace subtraction. These small input differences propagate to the final result.

16. What happens during the align() step if a project has no donors or a cluster has no members?

Zero-donor projects and zero-member clusters are dropped from the matrices. The functions still return a zero result for those projects so they appear correctly in the final output table.

17. How is the friendship matrix built inside COCM?

friendship_matrix = cluster_df.dot(cluster_df.transpose()) > 0. Two donors are considered friends if they share at least one cluster (i.e., they both donated to at least one common project when clusters = projects).

18. What do the k_indicators values actually mean?

For donor i and cluster g, k(i,g) is a number between 0 and 1 that represents how close donor i is to cluster g. 1 = direct member or fully connected through friends; 0 = completely unrelated through the friendship graph.

19. Is there a single “best” algorithm recommended for most quadratic funding rounds?

Most teams start with COCM + harsh=True for strong coordination resistance, or pctCOCM (≈ 0.3–0.6) when they want a gentle transition from classic QF. Pure QF remains the safest default when simplicity and maximum donor independence are paramount.

20. How can I exactly reproduce the legacy COCM results from older versions of the tool?

Select the “Legacy COCM” algorithm in the CSV calculator, or call legacy_COCM(donation_df, cluster_df) directly in Python. Make sure you use the same CSV preprocessing (no extra deduplication) and pass the original donation matrix as the cluster matrix.
